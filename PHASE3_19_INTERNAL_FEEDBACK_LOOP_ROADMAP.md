# PHASE3_19_INTERNAL_FEEDBACK_LOOP_ROADMAP.md

## Purpose

Define the path from Phase 3.18's deterministic rehearsal (PR #23,
catalog v0.26) to a **bounded internal feedback** prototype:
Pattern Ledger summaries — produced inside the processing window
— that re-enter ToyI's STREAM_APPEND path as deterministic
internal events, drive a *second-order* Pattern Ledger entry, and
emit normal Growth Ledger records. No new kernel surface, no LLM
call, no schema change, no consciousness-adjacent claim.

This roadmap is **research-and-engineering scaffolding**, not an
implementation order. The catalog patch plan in Step 5 fixes the
exact files, rows, fixtures, and version delta.

---

## 1. Where Phase 3.18 left ToyI

```text
external input
  -> _dispatch_stream_append
  -> _append_stream_chunk
      -> Pattern Ledger.observe              (first-order entry, recurrence += 1)
      -> Growth Ledger.observe(STREAM_CHUNK_ACCEPTED)
      -> Growth Ledger.observe(PATTERN_ENTRY_CREATED / UPDATED)
  -> _run_processing_window(seed_chunk)
      for k in 1..processing_window_size:
        _append_stream_chunk(text=seed_text, provenance="internal_processing_window:<k>:rehearsal")
        -> Pattern Ledger.observe          (same signature; recurrence += 1)
        -> Growth Ledger.observe           (more events)
```

Phase 3.18 thus produces a single Pattern Ledger entry whose
recurrence_count climbs deterministically with the rehearsal
count. ToyI **recognizes** but does not yet **feed back**: the
Pattern Ledger entry it just produced does not re-enter the
processing path as new evidence.

This is the gap Phase 3.19 closes.

---

## 2. Phase 3.19 target

Add a **bounded internal feedback** path that runs *inside*
`_run_processing_window` after each rehearsal (or after the
rehearsal sub-sequence, depending on Step 4 LOCK B), emitting a
deterministic Pattern Ledger summary chunk via the existing
`_append_stream_chunk` helper. Each summary chunk has:

```text
text       : deterministic Pattern Ledger summary string,
             e.g. "pattern_id=pledger:<id16> recurrence=<n>/256 sat=<state>"
provenance : "internal_processing_window:<k>:pledger_summary"
source     : OPERATOR (existing TextStreamSource.OPERATOR; no new
             source value introduced)
```

The summary text has a **different structural signature** from
the seed chunk (different len / distinct / repeat features), so
Pattern Ledger observes it as a **second-order** pattern entry
distinct from the seed entry. Recurrence climbs deterministically;
saturation eventually kicks in at the same 256 cap.

Architecturally:

```text
external input
  -> _dispatch_stream_append (seed -> first-order Pattern Ledger entry)
  -> _run_processing_window(seed_chunk)
      for k in 1..N:
        if feedback_mode == FEEDBACK_MODE_PATTERN_LEDGER:
            rehearsal_chunk = _append_stream_chunk(text=seed_text,
                                                   provenance=rehearsal_prov)
            -> updates first-order entry
            summary_chunk   = _append_stream_chunk(text=summary_text,
                                                   provenance=pledger_prov)
            -> creates / updates second-order entry
        else:
            (existing rehearsal-only behavior)
```

The summary text is generated by a new pure function
`build_pledger_summary_text(...)` in
`brain/development/processing_window.py`. Determinism:

```text
- Inputs: pattern_id (str), recurrence_count (int),
  saturation_state.value (str).
- Output: bounded printable string under STREAM_TEXT_MAX_LEN with
  no COGITO_ID inclusion, no raw chunk text inclusion, no time /
  PID / random / hostname / id() input.
- Output for the same inputs is byte-identical across runs.
```

Coherence Monitor feedback (architecture C) is **conditionally in
scope** — see Step 4 LOCK F. The default v1 ships only the
Pattern Ledger feedback path; Coherence summary may be bundled
behind a separate enum value (`COHMON_SUMMARY`, already reserved
in Phase 3.18) only if Step 4 unlocks it.

Combined feedback (architecture D) is **DEFERRED** unless Step 5
explicitly bundles it.

## 3. Distinguish modes

```text
A. rehearsal-only          (Phase 3.18 baseline, already shipped)
B. pattern-ledger feedback (Phase 3.19 v1 target)
C. coherence-monitor feedback         (conditional; see LOCK F)
D. combined pattern + coherence       (deferred)
E. future REPL / worldlet feedback    (deferred)
F. no-feedback / negative control     (rehearsal-only with
                                       feedback_mode=OFF)
```

All five branches share the same kernel; the difference is one
session-level bounded int / enum field plus a small number of
deterministic pure helpers in `brain/development/processing_window.py`.

## 4. Why this is a functional approximation of learning/growth

Bounded internal feedback is, structurally:

```text
- working-memory-like rehearsal (Phase 3.18 already)
- + a deterministic summary token of what was just recognized
- + that summary becomes a new input that the same ledger can
    recognize as a pattern in its own right
```

This pattern (input -> rehearsal -> summary -> summary becomes
input) is structurally similar to:

```text
- schema formation (Bartlett 1932; Rumelhart 1980): repeated
  exposures crystallize as a structured expectation
- conflict monitoring (Botvinick 2001): the system detects that
  it has seen X N times and emits a marker for that detection
- consolidation (McGaugh 2000): short-term traces are repeatedly
  re-presented and merge into long-term traces
- self-narration (Nelson & Fivush 2004): the system produces a
  description of its own recent activity which then re-enters
  perception
```

Use these only as architectural shorthand. **None of these
analogies licenses a consciousness, sentience, semantic, agency,
truth, or self-experience claim.** ToyI does not experience
recognition; it deterministically produces a record that it has
processed a recurrence.

## 5. Initial test window sizes

```text
N = 0   :  feedback path strictly OFF; verifies bit-identical
           parity with the Phase 3.18 rehearsal-only baseline
N = 5   :  short window; verifies the feedback chunk emerges
           the right number of times
N = 10  :  short window with a second saturation increment
N = 50  :  the "meaningful internalization" default carried
           forward from Phase 3.17; verifies that the runtime
           remains bounded under a real workload
```

`PROCESSING_WINDOW_SIZE_MAX = 255` is unchanged.
`50` remains the recommended initial high-window default for
internalization tests. The Phase 3.18 demonstration script
already exercises `N = 255` for saturation; Phase 3.19's
behavior report does the same with feedback turned on.

## 8. Hypothesis sketch (full set in Step 2 synthesis)

```text
H1: feedback generates inspectable second-order pattern entries
    beyond rehearsal.
H2: pattern-ledger feedback changes recurrence on the
    second-order entry while leaving the first-order entry's
    recurrence on the rehearsal cadence.
H3: coherence feedback, if implemented, creates a distinct trace
    without modifying truth claims.
H4: combined feedback has more inspectable structure than
    rehearsal-only.
H5: window size 50 shows distinct rehearsal vs feedback
    recurrence climbs.
H6: cache / call budget remains bounded; zero real model calls.
H7: no invariant or non-claim boundary is violated.
```
